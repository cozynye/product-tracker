## 1. 프로젝트 개요

### 서비스 목적
중고나라·번개장터의 중고 매물 시세를 자동으로 추적하는 1인용 모니터링 도구.
사용자가 등록한 검색어에 대해 주기적으로 크롤링을 수행하고, **적정가 이하 핫딜 매물**을 강조 표시하며 **Slack 알림**으로 빠르게 포착할 수 있게 한다.

### 기술 스택 요약

| 영역 | 기술 |
|------|------|
| 프론트엔드 | Next.js 15 (App Router), TypeScript, Tailwind CSS, shadcn/ui |
| 차트 | Recharts |
| 백엔드 / DB | Supabase (PostgreSQL) |
| 크롤링 | Cheerio (HTML 파싱), Axios |
| 알림 | Slack Incoming Webhook |
| 배포 | Vercel (프론트 + API Routes + Cron) |

### 배포 환경

- **프론트 + API**: Vercel (Hobby 플랜)
- **DB**: Supabase 클라우드 (`ylkvrvypcakbqiphmoqk.supabase.co`)
- **Cron 스케줄**: 매일 오후 3시 KST (Vercel Cron, Hobby 플랜 제한 — 일 1회)
- **알림 채널**: Slack `#중고-검색` 채널

### 아키텍처 구조 (요약)

```
사용자 브라우저
   ↓ 상세 페이지 진입
Next.js App Router (Vercel)
   ├─ /api/crawl/bunjang   → Supabase price_snapshots upsert
   ├─ /api/crawl/joonggonara
   └─ /api/cron/crawl-all  → 전체 monitor 자동 크롤링 (일 1회)

Supabase PostgreSQL
   ├─ user_monitors   (검색어 설정)
   └─ price_snapshots (수집된 매물)
```

## 3. 미비점 / 버그 (What's Missing or Broken)

### 🔴 Critical — 즉시 해결 필요

#### 3-1. 사용자 인증 없음
현재 Supabase Auth가 전혀 구현되어 있지 않다.
→ **현재 상태**: 1인 전용 서비스로 동작. DB 접근 제어가 없으므로 아무나 API를 호출하면 전체 데이터 열람·수정 가능.
→ **위험**: 서비스 URL이 공개되면 타인도 검색어·매물 데이터에 접근할 수 있음.

#### 3-2. Supabase RLS(Row Level Security) 미설정
DB 테이블에 Row Level Security 정책이 없어, anon key만 알면 전체 데이터 조회/삽입/삭제 가능.
→ 현재는 외부 노출 최소화로 운영 중이나, Vercel 환경변수 유출 시 DB 전체 위험.

#### 3-3. `/api/crawl/*` 인증 없음
`/api/crawl/bunjang`, `/api/crawl/joonggonara` 엔드포인트에 인증이 없음.
→ Cron API(`/api/cron/crawl-all`)와 달리 이 두 Route는 Bearer Token 검증 로직 누락.
→ 외부에서 `monitorId`를 추측하여 무단 크롤링 트리거 가능 (API rate limit 소진, DB 오염 위험).

---

### 🟡 기능 미비 — 단기 내 해결 권장

#### 3-4. 모니터(검색어) 수정 불가
등록된 검색어의 **키워드·카테고리 변경 UI가 없음**. 현재는 삭제 후 재등록만 가능.
→ `updateMonitor` Server Action은 구현되어 있으나, 키워드·카테고리 필드를 수정하는 다이얼로그 미구현.

#### 3-5. `alert_min_price` 컬럼 사실상 미사용
DB 스키마 및 TypeScript 타입(`IMonitor`)에 `alert_min_price` 필드가 존재하지만,
실제 알림 로직(`alert.ts`)에서는 `min_price`와 `max_price`만 사용하고 `alert_min_price`는 체크하지 않음.
→ 혼선 유발 가능. 컬럼 제거 또는 로직 통합 필요.

#### 3-6. 크롤링 실패 시 에러 알림 없음
크롤링 오류는 `console.error`로만 기록되고 Slack 등 외부 알림 없음.
→ Cron 실행 중 오류가 발생해도 운영자가 즉시 인지 불가.

#### 3-7. 번개장터 페이지네이션 미지원
번개장터는 1회 요청에 최대 100건만 수집 (`n=100`, page 고정).
→ 인기 검색어의 경우 100건 이후 매물 미수집 — 핫딜 누락 가능성 있음.

#### 3-8. 중고나라 크롤러 구조 변경 취약
중고나라는 HTML 파싱(Cheerio) 방식으로 구현됨. 사이트 HTML 구조 변경 시 크롤러 즉시 중단.
→ 번개장터(공식 API)와 달리 안정성 낮음.

---

### 🟢 UX 개선 필요 — 중기

#### 3-9. 수동 수집 결과 피드백 없음
상세 페이지 '수동 수집' 버튼 클릭 후 성공/실패/수집 건수 등 결과 표시가 없음.

#### 3-10. 알림 발송 이력 UI 없음
`price_snapshots.notified_at` 컬럼에 알림 발송 시각이 기록되지만, UI에 표시되지 않음.
→ 어떤 매물에 알림이 발송되었는지 확인 불가.

#### 3-11. 모바일 최적화 미검토
현재 대시보드·상세 페이지의 모바일 반응형 검토가 이루어지지 않음.

## 5. 기술 부채

| 항목 | 설명 | 해결 방향 |
|------|------|-----------|
| `alert_min_price` 컬럼 | DB + 타입에 존재, 로직에서 미사용 | 컬럼 제거 또는 알림 로직 통합 |

| `CRON_SECRET` 강도 부족 | 현재 16자 (`aB9kLmNp3RzT7xYq`) — 32자 이상 권장 | 시크릿 재생성 + Vercel 환경변수 업데이트 |

## 6. 요약

> **현재 상태**: 핵심 기능이 동작하는 1인용 MVP. 보안·확장성 강화 없이는 멀티 사용자 서비스로 전환 불가.

```
✅ 잘 동작하는 것       → 크롤링 자동화, Slack 알림, 핫딜 필터, 차트
⚠️  당장 위험한 것      → API 인증 누락, RLS 없음, 시크릿 강도
🔧 빠진 기능            → 모니터 수정, 알림 이력, 수집 결과 피드백
📅 중장기 필요한 것     → 다중 사용자, 페이지네이션, 크롤러 안정화
```

**다음 액션 권장 순서**:
1. `/api/crawl/*` 인증 추가 (1~2시간)
2. `alert_min_price` 컬럼 정리 (1시간)
3. 모니터 수정 다이얼로그 기획 + 개발 (0.5일)
4. Supabase Auth + RLS 도입 계획 수립 (별도 스프린트)
